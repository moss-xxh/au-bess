## 1. 代码溯源：这个平台做什么？

### 1.1 从代码推导项目定位

通过逆向分析代码库（28个HTML / 33,275行JS / 17,806行组件），按以下链条推导：

**证据链 ①：这是一个能源储能平台**
- 登录页标题：`<title>AU-03 - Australia Energy Storage</title>`
- 品牌标识：`design-system.css` 中出现 `X ENERGI`
- 登录页文案：`Virtual Power Plant — The Future of Distributed Energy`
- 电站数据模型包含：容量(MW/MWh)、荷电状态(SOC)、充放电功率、BMS/PCS/EMS设备

**推导 →** 项目属于能源储能（BESS）领域，自称虚拟电厂（VPP）。

**证据链 ②：面向澳洲NEM市场**
- 电站mock数据覆盖7个澳洲区域：`NSW, VIC, QLD, SA, WA, TAS, ACT`
- 代码中出现 `AEMO`（澳洲能源市场运营商）15次、`NEM` 4次、`FCAS`（频率控制辅助服务）6次
- 运行模式包含：`FCAS响应`、`套利模式`、`VPP调度`
- 调度面板支持按NEM区域切换

**推导 →** 目标市场是澳洲NEM（国家电力市场）。核心盈利模式是电力峰谷套利。FCAS出现在运行模式标签中，说明有FCAS设计意图，但代码中无任何FCAS业务逻辑实现。

**证据链 ③：存在两个业务角色，且权限严格分离**
- 登录后有 `role-select.html` 强制选择角色
- 两个角色：`owner`（业主）和 `operator`（运维方）
- Owner看7个电站（全部），Operator只看2个（代码：`stationData.slice(0, 2)`，注释写"模拟分配"）
- Owner有客户管理页面（`organization-new.html`），i18n中 "运维方" 出现12次
- Owner无调度权限，Operator有调度中心（dashboard）和操作日志

**推导 →** 业主拥有电站资产并分配给运维方管理。运维方负责实际调度操作，业主负责资产监控和客户（运维方）管理。这是一个**资产方-运营方分离**的商业模式。

**证据链 ④：核心业务是充放电套利**
- Dashboard核心功能：手动/自动充放电控制（`manualCharge`/`autoCharge` 等共152次出现）
- 有价格触发条件设置（`priceAbove`/`priceBelow`/`condition` 26次）、`spot price` 2次
- 有收益统计功能：`profit` 出现315次、`revenue` 107次、`收益` 33次
- 有时段配置（`timePeriods_v5`）用于设定充放电时间窗口

**推导 →** 运维方通过在低价时段充电、高价时段放电（峰谷套利）来盈利。自动调度模式还支持基于实时电价的条件触发。

**综合结论：**

> 本项目是一个**面向澳洲NEM市场的电池储能（BESS）虚拟电厂管理平台**。商业模式为：业主拥有储能电站资产 → 分配给运维方管理 → 运维方通过NEM市场峰谷套利盈利。FCAS出现在运行模式标签中，有设计意图但无实际实现。平台当前为纯前端Demo阶段，所有数据为模拟生成。

## 2. 商业模式推导：需要哪些域和功能？

从第1章的商业模式结论，逐步推导每个环节需要什么技术支撑。

### 2.1 第一环：业主拥有电站

**商业动作：** 业主购买/建设储能电站，需要登记和管理这些资产。

**推导出的域 → 资产管理域**

为什么？因为业主拥有多个电站（代码中有7个），每个电站有品牌（Tesla/BYD/Fluence）、型号（Megapack 2）、容量、位置、设备组成（BMS/PCS/EMS/变压器/空调/消防/电表/网关）。这些信息需要被记录、查询、维护。

**该域需要的功能：**
- 电站注册与基本信息（名称、位置、容量、设备型号）
- 设备台账（BMS/PCS/EMS等子设备清单及状态）
- 资产生命周期（投运日期、保修、衰减跟踪）

### 2.2 第二环：分配给运维方

**商业动作：** 业主把电站分配给运维方运营。代码证据：Owner看全部7个电站，Operator只看`slice(0,2)`（分配的2个），注释写"模拟分配"。

**推导出的域 → 用户权限域**

为什么？因为两个角色看到不同的数据、使用不同的功能。业主管资产和客户，运维方管调度和操作。代码中用 `localStorage.getItem('userRole')` 区分，导航菜单按角色过滤。

**该域需要的功能：**
- 角色定义（业主/运维方，未来可能有更多角色）
- 数据隔离（运维方只看自己被分配的电站）
- 身份认证（登录/注册/密码找回——代码中已有UI）
- 操作权限（业主不能调度，运维方不能管客户）

### 2.3 第三环：运维方需要看到电站状态

**商业动作：** 运维方要管理电站，首先得知道电站当前状态。代码证据：station-detail页面展示SOC、功率、温度、通讯状态；station列表有卡片视图和表格视图。

**推导出的域 → 实时监控域**

为什么？因为运维方需要实时看到每个电站的荷电状态(SOC)、当前功率、通讯在线/离线、运行模式。这些数据当前由Mock随机生成。

**该域需要的功能：**
- 电站状态总览（SOC、功率、通讯状态、运行模式）
- 设备级监控（BMS电压/温度、PCS功率、EMS指令）
- 通讯管理（在线/离线检测，代码中有commStatus: online/offline）
- 地图展示（代码中有经纬度坐标和卫星地图）

### 2.4 第四环：通过NEM市场套利

**商业动作：** 运维方需要知道当前电价才能决定何时充放电。代码证据：Dashboard有按NEM区域切换（NSW/VIC/QLD/SA/WA），有`spot price`字段，有`AEMO`出现15次。

**推导出的域 → 市场数据域**

为什么？套利的前提是知道"现在电价是多少"和"接下来电价会怎么变"。没有市场数据，充放电就是盲操作。

**该域需要的功能：**
- 实时电价获取（NEM 5分钟价格信号，按区域区分）
- 价格历史和趋势展示
- FCAS市场数据（代码有FCAS标签，虽然未实现，但设计意图在）

### 2.5 第五环：执行充放电操作

**商业动作：** 运维方在合适的时机下达充电/放电指令。代码证据：Dashboard有手动充放电按钮（`chargeBtn`/`dischargeBtn`），有自动模式（`autoCharge`/`autoDischarge`共152次），有条件触发设置。

**推导出的域 → 调度控制域**

为什么？这是核心盈利动作。代码中支持两种模式：手动操作（人点按钮）和自动模式（基于条件触发）。自动模式有价格阈值（`priceAbove`/`priceBelow`）、SOC阈值（`chargeStopSOC`/`dischargeStopSOC`）、时段配置（`timePeriods_v5`）。

**该域需要的功能：**
- 手动充放电指令（按钮 → 设备）
- 自动调度策略（价格条件、SOC条件、时段条件）
- 调度记录和执行日志（代码有dispatch-detail页面）
- 多区域独立调度（代码支持按区域切换调度面板）

### 2.6 第六环：计算盈利

**商业动作：** 运维方需要知道赚了多少钱。代码证据：`profit`出现315次、`revenue`107次、`收益`33次。有profit-new.html收益页面、todayRevenue/todayDischargeRevenue等字段。

**推导出的域 → 收益结算域**

为什么？套利的结果需要量化。买了多少度电、花了多少钱、卖了多少度电、赚了多少钱——这些都需要计算和展示。

**该域需要的功能：**
- 单次调度收益计算（充电成本 vs 放电收入）
- 日/月/年收益汇总
- 分电站收益统计
- 业主与运维方的收益分配（代码中Owner隐藏了部分利润列）

### 2.7 第七环：保障设备安全

**商业动作：** 储能电站有安全风险（过温、过压、通讯中断），需要及时发现和处理。代码证据：有fault-alarm.html告警页面、设备健康状态（ok/warn/error）、`告警`相关i18n 27条。

**推导出的域 → 告警域**

**该域需要的功能：**
- 实时告警触发（温度/电压/通讯异常）
- 告警列表和处理流程
- 告警升级机制
- 历史告警查询

### 2.8 第八环：运营数据分析

**商业动作：** 运维方需要回顾历史数据做决策优化。代码证据：有analysis-historical.html（295KB巨型文件）、reports.html报表页面、operation-log操作日志。

**推导出的域 → 报表分析域**

**该域需要的功能：**
- 历史数据趋势分析
- 运营KPI仪表盘
- 操作日志审计
- 数据导出

### 2.9 域总结

从商业模式8个环节，推导出8个必需的功能域：

| 商业环节 | 推导出的域 | 代码证据 |
|---|---|---|
| 业主拥有电站 | 资产管理域 | 7个电站、9类设备、品牌型号容量 |
| 分配给运维方 | 用户权限域 | owner/operator角色、slice(0,2)分配、菜单过滤 |
| 查看电站状态 | 实时监控域 | SOC/功率/通讯状态、地图、设备健康 |
| 获取市场电价 | 市场数据域 | NEM区域、AEMO 15次、spot price |
| 执行充放电 | 调度控制域 | 手动/自动模式152次、条件触发、时段配置 |
| 计算盈利 | 收益结算域 | profit 315次、revenue 107次、收益页面 |
| 保障安全 | 告警域 | fault-alarm页面、设备健康状态、27条i18n |
| 数据分析 | 报表分析域 | 历史分析页面、报表、操作日志 |

## 4. 对比：差距在哪？

### 4.1 逐域对比

| 域 | 第2章推导的需求 | 第3章实际现状 | 完整度 | 核心差距 |
|---|---|---|:---:|---|
| 资产管理 | 电站注册、设备台账、生命周期 | 有列表和详情UI，数据Mock | 30% | 无持久化，无生命周期管理 |
| 用户权限 | 角色、数据隔离、身份认证 | 角色存localStorage，菜单有过滤 | 15% | 无后端鉴权，可篡改 |
| 实时监控 | 遥测、通讯管理、实时刷新 | 有状态展示UI，数据随机 | 15% | 无真实数据源，无设备连接 |
| 市场数据 | NEM电价、价格趋势 | 有区域切换UI，无真实数据 | 5% | 零API对接 |
| 调度控制 | 手动/自动调度、策略引擎 | 有按钮和条件配置UI | 25% | 不连接设备，无执行通道 |
| 收益结算 | 套利计算、分成 | 有收益展示UI | 10% | 数字随机，无计算引擎 |
| 告警 | 实时触发、升级处理 | 有告警页面 | 15% | 数据Mock，无触发机制 |
| 报表分析 | 历史趋势、KPI、导出 | 有报表页面 | 15% | 数据随机，无导出 |

### 4.2 一句话总结

**8个域全部只完成了UI表现层（"看起来有"），数据层和逻辑层为零（"实际没有"）。平均完整度约16%。**

### 4.3 最致命的4个缺口

1. **无后端** — 所有数据存localStorage，刷新丢失，无法多人使用，无法持久化
2. **无设备连接** — 充放电按钮是死按钮，不连接任何真实设备
3. **无市场数据** — 不知道现在电价多少，套利决策无依据
4. **无安全性** — 角色可被篡改，任何人可冒充业主

## 5. 技术重构方案

### 5.1 为什么必须重构？（从代码层面）

不仅是功能缺失，代码架构本身也无法支撑后续开发：

**问题1：单文件膨胀**

| 文件 | 大小 | 问题 |
|---|---|---|
| dashboard.html | 349KB | 仍有~2000行内联JS |
| analysis-historical.html | 295KB | 巨型单文件 |
| js/dashboard-main.js | 488KB | 单个业务逻辑文件 |
| components/i18n.js | 326KB | 中英文翻译全部内联 |

**问题2：组件残留严重**
- `user-dropdown.js` / `user-dropdown-simple.js` / `user-dropdown-simple-new.js` / `user-dropdown-enhanced.js` — 4个版本共存
- `user-menu.js` / `user-menu-link.js` / `user-menu-simple.js` — 3个版本共存

**问题3：MPA架构的性能天花板**
- 28个独立HTML页面，每次导航 = 完整重新加载所有资源
- 无代码分割，无路由管理
- HTTP/1.1串行加载（已通过bundle缓解，但无法根治）

**问题4：无构建工具链**
- 无Vite/Webpack，无TypeScript，无ESLint
- 改一个组件要搜索28个HTML文件看哪里引用了
- 无法做tree-shaking、代码压缩、自动化测试

### 5.2 重构目标架构

```
[前端 - SPA]
Vue 3 + TypeScript + Vite + Tailwind CSS
    ├── Vue Router (单页路由, 替代28个HTML)
    ├── Pinia (全局状态管理, 替代localStorage)
    ├── Axios (API请求层)
    └── 组件库 (统一组件, 消除4个dropdown版本)
        
[后端 - API]
Node.js/Python FastAPI
    ├── JWT认证 + RBAC权限
    ├── REST API / WebSocket
    └── 设备协议适配 (OCPP/Modbus, 后期)

[数据层]
PostgreSQL + Redis
```

### 5.3 三阶段路径

**Phase 1：工程化翻新（1-2个月）**

### 5.4 现状 vs 目标

| 维度 | 现状 | 目标 | 为什么改 |
|---|---|---|---|
| 架构 | 28个独立HTML | Vue 3 SPA | MPA每次导航全量刷新，无法维护 |
| 数据 | localStorage随机Mock | PostgreSQL | 刷新丢数据，多人无法使用 |
| 权限 | `if(role==='owner')` | JWT + RBAC | 浏览器可篡改角色 |
| 组件 | 4个dropdown版本共存 | 统一Vue组件 | 复用率低，迭代残留 |
| 市场 | 零对接 | AEMO NEM API | 无真实电价，套利是盲操作 |
| 设备 | 零连接 | OCPP/Modbus | 充放电按钮是死按钮 |

## 6. 预期

### 重构后能实现什么

1. **从Demo变为产品** — 有真实数据、真实设备连接、真实套利收益
2. **多人使用** — 多个业主、多个运维方同时操作，数据隔离
3. **安全可靠** — JWT认证、RBAC权限、数据持久化
4. **快速迭代** — Vue组件化开发，改一处不影响其他
5. **规模化** — 从7个Mock电站到管理真实电站群
6. **真正盈利** — 对接NEM市场，实现真实的峰谷套利和FCAS服务

### 不重构会怎样

- 永远是Demo，无法
产生真实商业价值
- 每增加功能 = 创建新HTML + 复制粘贴，代码量线性增长，维护成本指数增长
- 无法对接真实设备和市场
- 无法多人协作使用
- 安全风险：任何人可篡改角色获取全部数据
